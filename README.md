# Fatsan Ore Generation

Fatsan Ore Generation, or `FUG`, is a Folia-first and Paper-compatible Minecraft `1.21.x` plugin focused on one job: changing ore generation without replacing terrain, cave, biome, or structure generation.

`FUG`, terrain/cave/noise/biome provider katmanlarini degistirmeden sadece ore davranisini ayarlamaya odaklanan, `1.21.x` icin Folia-oncelikli ve Paper-uyumlu bir eklentidir.

## Executive Summary

- Chosen architecture: a `BlockPopulator.populate(WorldInfo, Random, int, int, LimitedRegion)` ore post-processor attached during `WorldInitEvent`.
- Why: it preserves the original terrain/cave/structure/datapack pipeline and only mutates ore or host materials inside the worldgen region.
- Folia safety: the populator uses only `LimitedRegion` access, never mutates `World` directly, never schedules delayed block writes, and relies on immutable runtime rule snapshots.
- Paper compatibility: it uses stable Paper/Bukkit APIs only and no NMS.
- Scope: newly generated chunks only. Already generated chunks are not retroactively regenerated.

## Architecture Comparison

### Strategy A: `BlockPopulator` + `LimitedRegion`

- Folia safety: strong. Paper/Folia javadocs explicitly require thread-safe concurrent handling here, and `LimitedRegion` is the supported write surface.
- Paper safety: strong. This is standard API and does not replace base generation.
- Worldgen compatibility: strongest option because terrain, caves, structures, biome providers, noise settings, and datapack registries stay owned by the original generator.
- Performance: bounded scan over the central chunk’s ore band plus targeted vein placement.
- Maintenance: low to moderate, with no version adapter layer.

### Strategy B: custom `ChunkGenerator` wrapper/delegation

- Folia safety: possible, but much riskier because every generation phase must stay thread-safe.
- Compatibility: weaker. A wrapped generator still becomes part of the terrain/noise/cave pipeline and is easier to break when datapacks or custom generators evolve.
- Maintenance: high. It is more invasive and more fragile across generator changes.

### Strategy C: bootstrap/datapack/registry hybrid

- Folia safety: manageable, but complexity is much higher.
- Compatibility: potentially deep, but only worth it when a plugin must own registry-time worldgen definitions.
- Maintenance: highest. This would tie FUG to data pack ordering and registry lifecycle concerns that are unnecessary for ore-only control.

### Chosen Design

`FUG` uses Strategy A.

That choice matches the project’s hard goals best:

- ore-only control
- maximum compatibility with Terralith/Tectonic/Incendium-style packs
- zero terrain/noise/biome provider replacement
- no NMS
- clean Folia semantics through `LimitedRegion`

## Official API Research Notes

The design follows Paper/Folia’s own guidance:

- `BlockPopulator` javadoc recommends adding populators during `WorldInitEvent`, and the modern `populate(WorldInfo, Random, int, int, LimitedRegion)` method is documented as thread-safe and `LimitedRegion`-only.
- `LimitedRegion` javadoc warns not to read or write through `World` during generation because it can deadlock.
- `ChunkGenerator` javadoc describes ownership of terrain/noise/surface/caves/decorations/mobs/structures. Replacing or wrapping it would move FUG into much riskier territory than necessary.
- Folia docs explain regionized parallel ticking and why plugins must avoid stale cross-region access.

## Compatibility Model

FUG deliberately does **not**:

- replace the biome provider
- replace terrain noise
- replace cave carving
- replace structures
- replace the world’s chunk generator

FUG **does**:

- optionally rewrite already-generated vanilla ore blocks inside the target generation band
- place additional ore veins only into configured host materials
- stay conservative around non-host and unknown/custom materials
- leave unrelated blocks and decorators alone

This makes it suitable for worlds generated by vanilla, Paper-based generators, or datapacks such as:

- Terralith
- Tectonic
- Terratonic
- Incendium
- Amplified Nether
- similar custom generator/datapack stacks

## Folia and Paper Handling

- Internal plugin id: `FatsanOreGeneration`
- Display/branding name: `Fatsan Ore Generation`
- Reason: modern Paper/Folia rejects spaces in the `plugin.yml` `name` field
- `folia-supported: true` is declared in `plugin.yml`.
- FUG uses the Folia-safe `LimitedRegion` generation API.
- No async Bukkit/Paper world access is used in the generation path.
- No delayed block mutation is performed from worldgen callbacks.
- Runtime config is held as an immutable `RuleSnapshot` and read concurrently.

## Rule System

Inheritance order:

`global -> profile -> environment -> profile(named world) -> world -> biome`

Each ore rule supports:

- enable/disable
- rewrite-existing toggle
- generate-new-veins toggle
- frequency multiplier
- rarity
- spawn chance
- attempts
- attempts multiplier
- vein size min/max/multiplier
- irregularity and density
- vertical distribution profile
- host family restrictions
- allowed/forbidden host material lists
- air exposure bias
- realism bonuses/penalties
- optional raw block nodes for large veins

## Realism Features

Implemented realism controls include:

- mountain and badlands bonuses
- lush/dripstone cave bonuses
- deep dark and End penalties
- tuff-context bonus
- minimum solid-neighbor checks
- require-valid-host toggle
- air exposure penalty/boost/forbid/require modes
- host-family aware ore variant placement (`stone` vs `deepslate`)

Ancient debris is handled separately by default:

- own vertical profile
- strict air exposure behavior
- netherrack-only default host family
- smaller but rarer vein behavior

## Installation

1. Build with `./gradlew build`
2. Put the jar into `plugins/`
3. Start the server once so default configs are written
4. Edit `config.yml`, `worlds.yml`, and `ores.yml`
5. Use `/fug reload`

## What Changes and What Does Not

Changes:

- newly generated chunks
- ore presence/removal in those new chunks
- new ore vein placement in configured hosts

Does not change automatically:

- already generated chunks
- old regions that were created before FUG was installed
- existing chunk terrain, caves, or biome layout

If you need old terrain to reflect new ore settings, you need a separate regeneration/reprocessing workflow.

Eski chunk'lar otomatik olarak degismez. Yeni kurallar esas olarak yeni olusan chunk'lari etkiler.

## Known Limitations

- Because the plugin operates after base generation, it cannot recover the exact original host block for every already-generated vanilla ore block it removes.
- FUG only targets known vanilla ore-like resources by default. Unknown custom ore blocks are left untouched.
- End generation is conservative by default and effectively disabled unless explicitly configured.
- Runtime metrics are scaffolded in config but not yet wired to an external metrics backend.
